# v1.2.0 架构说明与分析

本文对 `v1.2.0` 架构方案做 GitHub 阅读友好的说明分析。完整设计见：[v1.2.0 架构方案](./v1.2.0-architecture-plan.md)。

## 背景判断

现有 v1.1.0 已经具备档期、订单、参与人提成和工资结算能力，但工资逻辑仍以“订单完成后一次性结算”为核心。

摄影工作室的真实经营场景通常不是这样：

- 定金、尾款、后期加选、产品补款可能发生在不同月份。
- 员工希望当月成交能反映到当月业绩。
- 公司不希望服务尚未完成时就发完全部提成。
- 退款、取消和异常交付需要可追溯的扣回机制。

因此，v1.2.0 的关键不是新增一个业绩页面，而是重构业绩和工资的底层计算口径。

## 关键架构判断

### 1. 必须新增收款账本

如果继续依赖订单金额字段，会出现一个根本问题：订单知道“应该收多少钱”，但不知道“每一笔钱实际在哪个月收到”。

所以“当月实际收款计入当月业绩”必须由独立收款账本支撑。

建议新增：

```text
bx_work_payment
```

该集合记录每笔收款、退款、冲减和调整，并保留业绩归属员工、团队、月份、来源和幂等键。

### 2. 必须新增提成事件账本

现有工资逻辑从订单参与人直接计算提成，适合简单结算，但不适合 70/30 冻结释放。

v1.2.0 推荐新增：

```text
bx_work_commission
```

将提成拆成事件：

```text
CURRENT
FROZEN
RELEASE
DEDUCT
ADJUST
```

这样才能解释每一笔工资从哪里来，也能处理退款扣回和冻结余额。

### 3. 工资单只做汇总，不做底层账本

`bx_work_payroll` 应继续作为工资发放记录，而不是承载全部提成拆分逻辑。

推荐链路：

```text
payment 收款账本
  ↓
commission 提成事件账本
  ↓
payroll 工资单汇总
```

这个结构比直接扩展工资表更稳定，也更容易审计。

### 4. 金额必须使用“分”

财务模块不能依赖浮点数。

所有核心金额字段建议使用整数分：

```text
PAYMENT_AMOUNT_CENT
COMMISSION_AMOUNT_CENT
PAYROLL_AMOUNT_CENT
```

展示时再转换为元。70/30 拆分应使用整数计算，尾差留给最后一笔或冻结部分修正。

### 5. 权限必须后端二次校验

免密码管理中心不等于弱权限。

管理员入口可以从“我的”页展示，但所有管理接口仍必须在云函数后端执行：

```text
openid -> staff -> STAFF_IS_ADMIN -> STAFF_STATUS
```

仅靠前端隐藏入口不具备安全性。

## 方案收益

### 对业务

- 业绩按实际收款月份计算，月度统计更可信。
- 70/30 规则能兼顾销售激励和交付约束。
- 退款、取消、补款、后期加选都有账可查。
- 工资明细能解释“先发、冻结、释放、扣回”的来源。

### 对工程

- 数据结构从订单状态升级为财务事件。
- 幂等键和状态机能降低重复发放风险。
- 收款、提成、工资职责分离，后续扩展更清晰。
- 开源项目更通用，不被某一个摄影工作室的配置写死。

## 主要风险

| 风险 | 影响 | 建议 |
|---|---|---|
| 只改页面不改账本 | 业绩和工资仍然会错 | 必须先做 payment / commission |
| 继续使用 float 金额 | 财务金额可能出现尾差 | 核心金额全部用 cent |
| 冻结金额无余额字段 | 退款后可能重复释放 | FROZEN 记录 remain / used |
| 发工资无幂等 | 重复点击可能重复发放 | payment、commission、payroll 都加 biz key |
| 只做前端权限 | 普通员工可能越权 | 管理接口后端校验 openid + admin |
| 旧订单无收款日期 | 历史业绩无法准确归月 | 先做补录清单，不自动乱迁移 |
| 云屿配置写死 | 开源复用成本高 | 工作室名称、云环境、规则配置化 |

## 实施优先级

建议按以下顺序推进：

1. 金额工具和账本模型。
2. 收款账本。
3. 提成账本。
4. 工资服务改造。
5. 业绩页。
6. 管理中心。
7. 历史数据兼容。
8. 全链路测试。

不要先做前端页面。前端页面如果先行，很容易被迫反复适配后端口径变化。

## 结论

v1.2.0 的方向是合理的，但它本质上是一次财务结构升级，不是普通 UI 迭代。

推荐采用：

```text
收款账本 + 提成事件账本 + 工资单汇总 + openid 管理员鉴权
```

这套方案能同时解决月度业绩、70/30 提成、订单完成释放、退款扣回、工资隐私和开源通用化问题。
