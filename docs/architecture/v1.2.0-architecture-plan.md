## 1. 需求合理性判断

### 1.1 需求整体合理

用户提出的规则：

```text
当月实际收到的钱，计入当月业绩。
对应提成当月先发 70%。
剩余 30% 暂时冻结。
订单完成之后，再结算冻结的 30%。
尾款、后续加选、产品等提成，也跟随订单完成之后结算。
```

该规则适合摄影工作室：

- 销售当月能看到成交成果，保持动力。
- 公司避免“钱刚收、服务还没交付，提成已经全部发完”。
- 30% 冻结和交付质量绑定，能降低服务后期摆烂风险。
- 对婚礼、宝宝宴、写真、活动跟拍这类有交付周期的摄影业务，合理。

### 1.2 不能直接按现有订单金额字段做

现有订单模型：

```text
cloudfunctions/mcloud/project/B00/model/work_order_model.js
```

已有字段：

```text
ORDER_AMOUNT
ORDER_DEPOSIT
ORDER_FINAL
ORDER_EXTRA
ORDER_ACTUAL_AMOUNT
ORDER_PARTICIPANTS
ORDER_PROGRESS
ORDER_SETTLE_STATUS
ORDER_COMPLETE_MONTH
```

问题：这些字段只能表示订单总金额、定金、尾款、加选金额，不能准确表示每一笔钱是什么时候收的。

所以：

> “当月实际收款计入当月业绩”必须新增 `bx_work_payment` 收款账本，不能只靠订单表。

### 1.3 当前工资逻辑必须重构

当前工资逻辑在：

```text
cloudfunctions/mcloud/project/B00/service/work_service.js
```

现有 `getPayrollForStaff` 是：

```text
查本月完成并待结算订单
  ↓
从 ORDER_PARTICIPANTS 里找参与人提成
  ↓
一次性进入工资
```

这适合旧版本“订单完成后一次性发工资”，不适合新版本“当月收款先发 70%，完成后释放 30%”。

所以本版必须新增：

```text
bx_work_payment       收款账本
bx_work_commission    提成账本
```

`bx_work_payroll` 继续作为最终工资发放记录，不直接承担底层提成拆分。

## 3. 核心业务规则

### 3.1 业绩口径

推荐：

```text
当月净业绩 = 当月有效收款金额 - 当月退款 / 冲减金额
```

收款包括：

```text
定金 deposit
尾款 final
后期加选 extra
产品 product
补款 supplement
手动调整 adjust
```

退款 / 冲减包括：

```text
订单取消退款
服务异常退款
改价冲减
人工调整
```

排行榜默认使用：

```text
净业绩
```

管理员可以切换：

```text
毛收款
退款冲减
净业绩
```

普通员工默认只看净业绩和排行，不看别人详细工资。

### 3.2 业绩归属

推荐规则：

```text
优先取订单参与人中的“销售”角色。
如果没有销售，则取订单创建人。
如果仍没有，则进入“未归属收款”待管理员处理。
```

必须在收款账本中保存快照：

```text
PAYMENT_STAFF_ID
PAYMENT_STAFF_NAME
PAYMENT_TEAM_ID
PAYMENT_TEAM_NAME
```

原因：员工后续改名、调团队、离职，不应该影响历史业绩归属。

### 3.3 提成口径

推荐主规则：

```text
订单未完成 / 未审核通过前产生的收款：
  提成 70% 进入当月可发
  提成 30% 进入冻结

订单完成并审核通过后：
  释放该订单历史冻结的剩余 30%

订单完成审核通过后新增收款：
  该笔提成 100% 进入当月可发
```

这比“所有收款都先 70/30”更符合用户原话，因为用户强调订单完成之后再结算冻结的 30%，尾款、加选、产品等也跟随订单完成之后结算。

### 3.4 固定提成和手动提成

现有系统支持：

```text
percent 按比例
fixed 固定金额
manual 手动金额
none 不计提成
```

推荐规则：

| 模式 | v1.2.0 规则 |
|---|---|
| percent | 每笔收款直接按比例算提成 |
| fixed | 按订单收款进度分摊固定提成 |
| manual | 按订单收款进度分摊手动提成 |
| none | 不生成提成 |

固定 / 手动提成不能每笔收款都全额发，否则会重复发。

推荐算法：

```text
订单固定提成总额 = 1000
订单应收金额 = 10000

某次收款 = 3000
本次分摊比例 = 3000 / 10000
本次固定提成 = 1000 * 30%
```

最后一笔收款时做尾差修正，保证总提成不超过固定金额。

### 3.5 退款和取消规则

必须做，否则会错账。

推荐规则：

```text
当月退款：
  冲减当月业绩

对应提成：
  优先冲减未发 70%
  再冲减未释放冻结 30%
  冻结不够时，生成工资扣回项
  已经发过的 70%，进入当月工资扣回
```

关键修正点：

> 冻结提成必须有“剩余可释放金额”，不能只记录原始冻结金额。  
> 如果退款已经抵扣了冻结金额，后续订单完成时不能再把原冻结 30% 全额释放。

否则会出现：

```text
冻结 150
退款先抵扣冻结 150
订单完成又释放 150
= 多发 150
```

所以 `FROZEN` 行必须有：

```text
COMMISSION_AMOUNT_CENT
COMMISSION_FROZEN_REMAIN_CENT
COMMISSION_FROZEN_USED_CENT
```

释放时只释放：

```text
COMMISSION_FROZEN_REMAIN_CENT
```

### 3.6 订单完成定义

不建议把“拍摄完成”直接等同于“订单完成”。

推荐完成条件：

```text
服务已完成
尾款 / 应收状态已确认
管理员审核通过
```

现有订单状态：

```text
ORDER_PROGRESS = 40 已完成
ORDER_SETTLE_STATUS = 10 待审核
ORDER_SETTLE_STATUS = 30 待结算
ORDER_SETTLE_STATUS = 40 已结算
```

推荐挂接点：

```text
admin_work_service.auditOrder 审核通过后
  ↓
释放冻结提成
  ↓
进入当月工资可发项
```

现有位置：

```text
cloudfunctions/mcloud/project/B00/service/admin/admin_work_service.js
```

## 4. 金额精度规则：必须用“分”

现有 `_money()` 使用 float：

```text
Math.round(val * 100) / 100
```

旧页面展示可以继续用，但新账本不能依赖 float。

v1.2.0 财务模块必须新增：

```text
cloudfunctions/mcloud/project/B00/service/work_money_util.js
```

建议方法：

```js
moneyToCent(value)
centToMoney(cent)
splitCent(totalCent, currentRatePermille)
assertCent(cent)
```

核心金额字段必须使用 int：

```text
PAYMENT_AMOUNT_CENT
COMMISSION_AMOUNT_CENT
PAYROLL_AMOUNT_CENT
```

展示时再转成元。

70/30 拆分推荐：

```text
currentCent = floor(totalCent * 700 / 1000)
frozenCent = totalCent - currentCent
```

这样不会丢分。

### 6.2 收款服务：`work_payment_service.js`

职责：

```text
新增 / 修改未锁定收款
作废未结算收款
新增退款 / 冲减记录
同步订单金额到收款账本
更新订单收款摘要
写财务日志
```

关键方法：

```js
savePayment(openId, input)
voidPayment(openId, paymentId, reason)
listPayments(openId, query)
syncOrderPayments(order, oldOrder)
buildPaymentSummary(orderId)
```

重要规则：

```text
未进入工资的 payment 可以编辑。
已生成 paid commission 或 payroll 的 payment 不能覆盖。
修改历史金额必须生成 adjust/refund。
```

### 6.3 提成服务：`work_commission_service.js`

职责：

```text
根据 payment 生成 commission
70/30 拆分
订单完成后释放冻结
退款扣回
固定 / 手动提成分摊
提成幂等
```

关键方法：

```js
generateByPayment(paymentId)
releaseFrozenByOrder(orderId, releaseMonth)
deductByRefund(paymentId)
previewCommissionForPayment(payment)
```

伪流程：

```text
for payment in effective payments:
  if payment.amount == 0:
    skip

  找订单
  找参与人
  计算每个参与人的总提成

  if 订单已完成并审核通过:
    生成 CURRENT 100%
  else:
    生成 CURRENT 70%
    生成 FROZEN 30%

  保存规则快照
  写入幂等键
```

冻结释放：

```text
查该订单所有 FROZEN 行
只释放 COMMISSION_FROZEN_REMAIN_CENT > 0 的部分
生成 RELEASE 行
更新 FROZEN 行 remain / used / status
```

### 6.4 业绩服务：`work_performance_service.js`

职责：

```text
个人业绩
团队业绩
全店业绩
排行榜
```

## 12. 并发、幂等和审计

财务相关动作必须是 v1.2.0 必做，不应放后续版本。

### 12.1 必须幂等的动作

```text
保存收款
生成提成
释放冻结
退款扣回
工资发放
历史迁移
```

每个动作都要有：

```text
BIZ_KEY
```

例如：

```text
payment:orderId:type:date:amount:index
commission:paymentId:staffId:role:kind
release:orderId:frozenCommissionId:releaseMonth
payroll:staffId:month
```

### 12.2 必须使用事务 / 锁的动作

```text
收款保存 + 提成生成 + 订单摘要更新
审核通过 + 冻结释放
工资发放 + commission 锁定 + payroll 创建
退款 + 提成扣回
```

不能继续依赖：

```text
先 count
再 insert
再判断 count
```

这种方式在双击、重试、云函数并发时会重复发放。

## 17. 最终推荐方案

推荐采用：

```text
收款账本 + 提成事件账本 + 工资单汇总 + openid 管理员鉴权
```

而不是：

```text
继续扩展订单表金额字段
继续从 ORDER_PARTICIPANTS 一次性算工资
继续复用旧密码后台 token
```

原因：

1. 能准确按月份统计业绩，因为每笔收款都有 `PAYMENT_DATE` 和 `PAYMENT_MONTH`。
2. 能处理 70/30 冻结释放，因为提成被拆成 CURRENT / FROZEN / RELEASE / DEDUCT。
3. 能防止错账和重复发放，因为 payment、commission、payroll 都有幂等键和状态。
4. 能处理退款和取消，因为退款是负向 payment，扣回是 DEDUCT commission。
5. 能保护工资隐私，排行榜只展示业绩，工资明细只给本人和管理员。
6. 能兼容现有代码结构，不推翻 B00 项目，只是在现有 model/service/controller/page 结构下扩展。
7. 能保持开源通用性，云屿摄影作为案例，底层仍是通用摄影工作室业务框架。

## 20. 最终结论

这次升级需求方向是对的，但不能只做页面。

**必须从订单金额字段升级为“收款账本 + 提成账本”的财务结构。**

最稳的 v1.2.0 方案是：

```text
底部新增业绩 tab
新增 payment 收款账本
新增 commission 提成账本
70/30 拆分
订单审核通过释放冻结余额
工资从 commission 汇总
管理员从我的页基于 openid 进入管理中心
所有管理接口后端二次校验
旧后台保留兼容但不再作为日常入口
```

这份架构已经可以作为后续真正编码的实施蓝图。下一步应继续生成“逐文件无脑执行编码施工图”。
