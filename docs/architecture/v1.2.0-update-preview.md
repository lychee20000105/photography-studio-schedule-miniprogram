# v1.2.0 更新预告：业绩、提成、工资与免密码管理中心

> 状态：架构预告 / 实施规划  
> 适用项目：摄影工作室档期小程序  
> 当前稳定代码版本：v1.1.0  
> 规划目标版本：v1.2.0

v1.2.0 计划把现有“档期 + 订单 + 员工提成 + 工资结算”能力，升级为更适合摄影工作室真实经营的业绩与财务协同模块。

本次预告不是代码发布说明，而是下一轮开发的架构方向。完整方案见：[v1.2.0 架构方案](./v1.2.0-architecture-plan.md)。

## 一句话概括

v1.2.0 将新增“收款账本 + 提成账本 + 工资单汇总”的财务结构，并让管理员从小程序“我的”页面免密码进入管理中心。

## 计划新增能力

### 1. 底部导航新增业绩入口

当前底部导航中的“小记”将调整为“业绩”。

员工进入后可以查看：

- 本月个人业绩
- 本月排名
- 业绩明细
- 提成摘要
- 待释放提成

“小记”能力不删除，后续可从“新增”或“我的”入口保留。

### 2. 新增收款账本

现有订单字段只能描述订单总金额、定金、尾款和后期金额，不能准确表达“哪一天实际收到多少钱”。

v1.2.0 计划新增 `bx_work_payment` 收款账本，用每一笔收款、退款、补款和冲减记录支撑业绩统计。

核心口径：

```text
当月净业绩 = 当月有效收款金额 - 当月退款 / 冲减金额
```

### 3. 新增提成账本

v1.2.0 计划新增 `bx_work_commission` 提成事件账本，不再直接从订单参与人一次性计算工资。

提成事件将拆分为：

- `CURRENT`：当月可发
- `FROZEN`：冻结部分
- `RELEASE`：完成后释放
- `DEDUCT`：退款或异常扣回
- `ADJUST`：人工调整

### 4. 支持 70/30 提成规则

计划规则：

```text
订单未完成前产生的收款：
  70% 提成进入当月可发
  30% 提成进入冻结

订单完成并审核通过后：
  释放剩余冻结提成

订单完成后新增收款：
  该笔提成 100% 进入当月可发
```

该规则适合婚礼、宝宝宴、写真、活动跟拍等有交付周期的摄影业务。

### 5. 工资页改造

工资不再只展示“订单完成后一次性结算”的结果，而是解释清楚：

- 当月先发提成
- 本月释放提成
- 退款扣回
- 待释放冻结金额
- 已发 / 待发 / 失败状态

### 6. 小程序内管理中心

管理员将可以从“我的”页面进入管理中心，不再依赖旧后台密码入口作为日常操作方式。

权限仍必须由后端基于 `openid + STAFF_IS_ADMIN` 二次校验，前端入口隐藏不等于权限安全。

### 7. 排行榜和隐私控制

普通员工可以查看排行，但不暴露其他员工详细工资。

默认展示：

- 排名
- 员工脱敏信息
- 净业绩
- 自己的明细

详细工资、扣回、发放状态仅本人和管理员可见。

## 暂不纳入 v1.2.0 的范围

为控制开发风险，v1.2.0 不建议同时加入以下能力：

- 多门店 SaaS
- 微信支付自动对账
- 电子合同
- 阶梯提成
- 主管分润
- 团队奖金
- 完整旧后台迁移
- 复杂绩效体系

这些能力更适合在财务账本稳定后逐步扩展。

## 对使用者的影响

### 对摄影工作室

- 更容易按“实际收款月份”统计业绩。
- 能减少订单未交付前提成全部发完的风险。
- 能把退款、取消、补款、后期加选纳入统一账本。
- 管理员可直接在小程序内处理关键经营数据。

### 对员工

- 当月成交能及时看到业绩反馈。
- 提成来源更透明。
- 冻结和释放规则可解释。
- 工资明细不会被其他普通员工看到。

### 对二次开发者

- 需要从“订单金额字段”思路升级为“财务事件账本”思路。
- 需要重视金额精度、幂等、事务、审计和权限。
- 不建议只改页面，因为核心问题在数据结构和后端结算链路。

## 当前建议实施顺序

1. 冻结 v1.2.0 范围和业务规则。
2. 新增金额工具，金额统一使用“分”为单位。
3. 新增 `payment`、`commission`、`finance_log` 模型。
4. 扩展订单、员工、工资模型。
5. 实现管理员 openid 鉴权。
6. 实现收款账本。
7. 实现提成账本和 70/30 拆分。
8. 改造工资服务。
9. 新增业绩页和管理中心。
10. 做历史数据兼容和测试。
