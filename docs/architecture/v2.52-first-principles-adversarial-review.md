# v2.52 第一性原理架构与对抗式审查

> 日期：2026-07-05  
> 范围：云屿摄影档期小程序阶段性架构基线，重点覆盖微信小程序前端、`mcloud` 云函数、工作台业务、财务/工资/审核、小猫 Agent、live patch 覆盖层和版本发布链路。  
> 结论：当前 v2.51 可作为小猫 MiMo 修复后的运行基线；v2.52 修复了小猫首次会话回复显示问题，并可作为“工作台 + 小猫 Agent + 财务/工资/审核 + 发布边界”的阶段性架构基线。预约、内容、样片、拍摄服务、用户数据和传统后台等完整程序模块需要后续单独补齐。

## 1. 第一性原理

这个小程序最底层要保护的不是“页面能打开”，而是四类事实：

1. 档期事实：某一天有哪些拍摄、订单、事项、休息和备注。
2. 经营事实：订单、客户、收款、退款、提成、工资、审核状态必须可追溯。
3. 权限事实：员工、管理员、审核人、AI Agent 都只能在自己权限范围内读写。
4. 发布事实：每次改动必须能回滚、能解释、能验证、能区分开发版上传和正式发布。

因此架构原则是：

- 前端只负责交互、展示、输入收集和低风险本地状态。
- 云函数负责所有业务事实、权限判断、金额计算、AI Key 使用和审计流水。
- 数据库集合是事实来源，不能让聊天记录、页面缓存或模型回复替代数据库。
- AI Agent 只能通过后端白名单工具写入业务数据，不能凭自然语言直接改高风险数据。
- 版本号、日志、开发版上传、Git checkpoint 和 push 必须共同形成可回滚链路。

## 2. 当前架构事实

项目规模：

- 页面配置文件：91 个；`app.json` 注册页：93 个。
- 自定义组件：26 个。
- 云函数目录：2 个，核心为 `mcloud`，另有 `autoSendDaily`。
- `mcloud` controller：23 个。
- `mcloud` service：30 个。
- `mcloud` model：23 个。

运行链路：

```text
用户操作
  -> miniprogram 页面 / work_pet 组件
  -> helper/cloud_helper.js
  -> wx.cloud.callFunction({ name: "mcloud", route, PID, token, params })
  -> framework/core/application.js
  -> project/B00/public/route.js
  -> controller
  -> service
  -> model / setupUtil / cloud storage / external API
  -> 统一 code/data/msg 返回前端
```

关键入口：

- 工作台首页：`projects/B00/pages/work/calendar/work_calendar`
- 小猫组件：`miniprogram/cmpts/work_pet`
- 云函数入口：`cloudfunctions/mcloud/index.js`
- 路由表：`cloudfunctions/mcloud/project/B00/public/route.js`
- 工作台业务：`work_controller.js`、`work_service.js`
- 管理中心业务：`work_admin_controller.js`、`admin_work_service.js`
- 传统后台业务：`admin/*` 路由，使用后台管理员 token；小程序内管理中心 `work/admin_*` 使用员工 openid + `STAFF_IS_ADMIN`，两套路由入口不能混为一套权限模型。

- AI 确认队列：`work_agent_confirm_service.js`
- AI 审计流水：`work_agent_audit_model.js`

## 3. 模块边界

### 前端层

前端负责：

- 页面路由、表单、列表、筛选、弹窗、聊天 UI。
- 调用 `cloudHelper`，不直接调用第三方 AI API。
- 小猫本地对话、多会话、本地图片预览、发送按钮状态。
- 只做用户体验兜底，不把兜底文案当作真实 API 成功。

前端不能负责：

- 保存原始 API Key。
- 直接计算最终工资、提成、退款影响。
- 绕过云函数改订单、收款、工资、审核。
- 把模型回复直接写成业务事实。

### 云函数层

云函数负责：

- 路由分发、参数校验、权限校验。
- 订单、事项、休息、备注、消息、绩效、工资、审核。
- AI 供应商配置、Key 使用、模型请求、usage 回传。
- 高风险动作确认队列和审计流水。

云函数必须保持：

- 所有写操作都经过 controller validateData 和 service 权限判断。
- 高风险动作保留可追溯记录。
- 金额用后端统一口径计算。
- 外部 AI 不可用时返回明确失败，不能伪装成模型回复。

### 数据层

核心集合：

- `bx_work_staff`：员工、绑定、管理员身份。
- `bx_work_order`：订单和档期主体。
- `bx_work_item` / `bx_work_rest` / `bx_work_note`：事项、休息、小记。
- `bx_work_payment`：收款/退款/冲减账本。
- `bx_work_commission`：提成事件账本。
- `bx_work_payroll`：工资汇总和发放记录。
- `bx_work_finance_log`：财务变动日志。
- `bx_work_agent_audit`：AI 动作审计。
- `bx_work_agent_confirm`：高风险 AI 动作确认队列。

数据层原则：

- 订单是业务主体，不是唯一财务账本。
- 收款、提成、工资必须分层记录，便于追溯和冲正。
- 审核、作废、发放类动作必须有状态和日志。
- 客户隐私、手机号、Key、Token 不进入版本日志或公开文档。

## 4. 小猫 Agent 边界

小猫的正确闭环：

```text
work_pet 输入
  -> work/ai_chat
  -> WorkAiService.assertStaff
  -> 读取云端 AI providers 配置
  -> 请求 MiMo / 其他兼容供应商
  -> 解析回复和 JSON action
  -> 低风险动作直接后端校验执行
  -> 高风险动作进入确认队列
  -> 写入审计流水
  -> 前端刷新页面
```

已验证事实：

- v2.51 后 `work/ai_chat` 真实返回 `MiMo` / `mimo-v2.5`。
- 部署后复测标记：`OK0705-YYX8PK`。
- 部署后复测 `usage.total_tokens=774`。
- `aiUnavailable`、本地快捷回复、配置读取成功都不能算 API 成功。

Agent 高风险动作：

- 取消订单。
- 录入或作废收款。
- 发放工资。
- 审核订单。

这些动作必须保留人工确认或管理员后端校验，不应让模型直接决定最终结果。

## 5. 对抗式审查结论

本轮采用主代理实现和多代理只读对抗审查结合的方式：

- Heisenberg：重点审查版本口径、发送按钮、MiMo 真实性、敏感信息和发布基线。
- Kuhn：重点审查 v2.52 架构文档是否过度声明完整程序覆盖、风险边界和后续基线条件。
- Lovelace：重点审查完整调用链、live patch 覆盖层、双后台入口、AI 动作风险分级和财务一致性风险。
- 主代理：整合审查结论并执行版本、上传、提交、push、finish 收尾。

### 已满足

- 最近版本记录已经改成中文：`v2.51`、`v2.50`、`v2.49` 在 `miniprogram/version.js` 中均为中文记录。
- 小猫发送按钮不再使用 `&#x2191;`，当前 WXML 使用中文 `发送`。
- MiMo 不是兜底成功：已通过 DevTools automator 真实调用云函数，拿到真实 provider/model/usage。
- API Key 没有写入本地源码、日志或提交。
- v2.51 已完成云函数 live patch 部署、微信开发版上传、Git commit、GitHub push。

### 非阻塞风险

1. 历史文件仍有编码债务。  
   `work_ai_service.js` 等文件仍有历史乱码，v2.51 只修了本次影响小猫可用性的提示词和用户可见记录。

2. `work_ai_service.js` 职责过重。  
   该文件同时处理配置、模型请求、提示词、工具解析、动作执行、审计和兜底。后续应拆成配置、请求、工具执行、审计四层。

3. live patch 是运行时覆盖层。  
   本轮已校验 `work_ai_service_live_patch.js` 解压后与源码一致；后续任何云函数逻辑修改都必须同步校验源码与 live patch，否则可能出现“看源码是新逻辑、云端运行旧逻辑”的问题。

4. 高风险动作跨服务依赖复杂。  
   收款、提成、工资、审核已经有保护，但后续任何改动都必须从账本一致性和回滚路径出发，不宜只改页面。

5. 旧版架构文档有历史乱码和旧路径。  
   `docs/architecture/v1.2.0-*` 仍保留历史规划痕迹，后续可归档或重写，但不应在本轮和业务修复混在一起。

6. 开发版上传不等于正式发布。  
   当前自动化只覆盖开发版上传和云函数部署，不代表微信审核通过或正式线上发布。

7. 发布基线必须以干净工作区为准。  
   对抗审查发现 v2.52 中间态存在未提交文件和 pending 口径，因此本轮 finish 前必须完成验证、开发版上传、提交、push，并把日志改为完成态。

### 不建议本轮直接改代码的事项

- 全量修复 `work_ai_service.js` 历史乱码。
- 拆分 AI service 大文件。
- 重构财务/工资/审核账本。
- 迁移旧架构文档。
- 改云数据库集合结构。

这些都是中高风险改动，需要独立版本、独立验收和真实业务数据保护方案。

## 6. 后续架构改造优先级

P0：保持当前可用基线

- 保持小猫 `work/ai_chat` 云函数链路。
- 每次 AI 相关修复必须拿真实 provider/model/usage。
- 继续区分兜底文案和模型回复。

P1：AI 服务分层

- `work_ai_config_service`：providers、Key、模型、Base URL。
- `work_ai_provider_client`：OpenAI-compatible 请求、MiMo 兼容、usage 解析。
- `work_ai_tool_service`：JSON action 解析和白名单。
- `work_ai_audit_service`：审计、确认队列、失败记录。

P2：编码债务治理

- 先建立只读扫描报告。
- 按文件职责分批恢复中文。
- 每批修复必须运行 U+FFFD 扫描、JS syntax check、开发版上传和真实关键链路测试。

P3：文档治理

- 保留 v1.2.0 历史规划，但标注旧路径/旧编码风险。
- 将当前运行架构以 v2.x 文档为准。
- 后续重大模块改造都补充 architecture decision record。
